╔══════════════════════════════════════════════════════════════════════════════════════╗
║  STEP 1 — EMAIL INTAKE AND PRE-PROCESSING                                            ║
╚══════════════════════════════════════════════════════════════════════════════════════╝

  [VENDOR]
     │
     │──► Sends email to vendor support mailbox
     │
  [MAILBOX / EMAIL SYSTEM]
     │
     │──► DETECT new inbound email in vendor support mailbox
     │
     │──► CAPTURE metadata:
     │         ├── Sender Email Address
     │         ├── Subject Line
     │         ├── Timestamp
     │         ├── Thread Identifiers  ◄── CRITICAL for linking replies
     │         └── Attachments
     │
     │──► SEND email body + all metadata ──► to GenAI Engine
     │
     │
  📌 WHY THIS MATTERS:
     Thread identifiers are crucial so replies can be
     linked to the right existing ticket automatically.
     This step converts raw email into a structured request.


╔══════════════════════════════════════════════════════════════════════════════════════╗
║  STEP 2 — GenAI UNDERSTANDING (INTENT + EXTRACTION)                                  ║
╚══════════════════════════════════════════════════════════════════════════════════════╝

  [GenAI ENGINE]
     │
     │──► INTENT CLASSIFICATION
     │         ├── Payment Delay
     │         ├── Invoice Clarification
     │         ├── Contract Query
     │         ├── Technical Issue
     │         ├── Access Issue
     │         └── Compliance Docs
     │
     │──► ENTITY EXTRACTION
     │         ├── Vendor Name
     │         ├── Vendor ID (if present in email)
     │         ├── Invoice / Reference Numbers
     │         ├── Dates
     │         └── Requested Action
     │
     │──► URGENCY DETECTION
     │         ├── Keywords: 'urgent', 'escalation', 'immediately'
     │         └── Plus Business Rules applied
     │
     │──► SENTIMENT ASSESSMENT
     │         ├── Angry / Frustrated  ──► Flag for escalation
     │         └── Neutral             ──► Normal flow
     │
     │──► CONFIDENCE SCORE GENERATED
     │         │
     │         ├── HIGH CONFIDENCE ──────────────────────► AUTO PROCESS
     │         │                                            Continue to Step 3
     │         │
     │         └── LOW CONFIDENCE ──────────────────────► HUMAN TRIAGE QUEUE
     │                                                     Stop auto processing
     │                                                     Route to Support/Triage
     │                                                     agent for manual review
     │
  📌 CONTROL RULE:
     If confidence score is below threshold →
     Route email to human triage queue BEFORE
     creating or sending anything automatically.


╔══════════════════════════════════════════════════════════════════════════════════════╗
║  STEP 3 — VENDOR MATCH AND CONTEXT FETCH (SALESFORCE)                                ║
╚══════════════════════════════════════════════════════════════════════════════════════╝

  [ORCHESTRATION LAYER + SALESFORCE]
     │
     │──► MATCH ATTEMPT 1: Email Match
     │         └── Match sender email ──► Vendor Contacts in Salesforce
     │                   │
     │                   ├── MATCH FOUND ──────────────────► Proceed to Context Fetch
     │                   │
     │                   └── NO MATCH ──────────────────────► Try Match Attempt 2
     │
     │──► MATCH ATTEMPT 2: Vendor ID Match
     │         └── Use Vendor ID extracted by GenAI (if available)
     │                   │
     │                   ├── MATCH FOUND ──────────────────► Proceed to Context Fetch
     │                   │
     │                   └── NO MATCH ──────────────────────► Try Match Attempt 3
     │
     │──► MATCH ATTEMPT 3: Name Similarity Match
     │         └── Use name similarity algorithm WITH safeguards
     │                   │
     │                   ├── MATCH FOUND ──────────────────► Proceed to Context Fetch
     │                   │
     │                   └── STILL NOT FOUND ───────────────► Route to Human Triage
     │
     │──► CONTEXT FETCH from Salesforce (once matched):
     │         ├── Vendor Status     (Active / Inactive / Blocked)
     │         ├── Vendor Tier / Criticality
     │         ├── Contract References
     │         ├── SLA Category
     │         └── Escalation Contacts
     │
  📌 WHY CRITICAL:
     Wrong vendor match can cause DATA LEAKAGE and
     INCORRECT COMMUNICATION. Matching must be
     multi-step and conservative.

╔══════════════════════════════════════════════════════════════════════════════════════╗
║  STEP 8 — HOLD / RESUME LOGIC                                                        ║
╚══════════════════════════════════════════════════════════════════════════════════════╝

  [ServiceNow + Orchestration Layer]
     │
     │──► SCENARIO A: Documents needed from Vendor
     │         │
     │         └──► SET ticket status: "Awaiting Vendor"
     │               └──► SEND Document Request Email to Vendor (GenAI Draft)
     │               └──► SLA PAUSED ◄── Vendor's turn now
     │
     │──► SCENARIO B: Internal dependency exists
     │         │       (e.g., finance team confirmation needed)
     │         └──► SET ticket status: "Awaiting Internal Info"
     │               └──► SLA PAUSED ◄── Internal team's turn
     │
     │──► SLA PAUSE/RESUME rules:
     │         ├── Applied as per policy
     │         └── Documented in ServiceNow
     │
     │──► RESUME trigger:
     │         ├── Vendor sends required documents ──► Resume SLA, update ticket
     │         └── Internal team confirms ───────────► Resume SLA, update ticket
     │
  📌 WHY IMPORTANT:
     Hold states prevent SLA from unfairly breaching
     when the ball is in the vendor's court.
     Must be governed consistently.

╚══════════════════════════════════════════════════════════════════════════════════════╝

  [Orchestration Layer + ServiceNow + GenAI]
     │
     │──► MONITOR SLA Timers continuously
     │         │
     │         ├── 70% SLA consumed  ──────────────────► ALERT triggered (Internal)
     │         ├── 85% SLA consumed  ──────────────────► ESCALATION WARNING
     │         └── 95% SLA consumed  ──────────────────► CRITICAL ESCALATION
     │
     │──► AUTO ESCALATION triggers:
     │         ├── High-tier vendor detected ──────────► Auto Escalate
     │         └── Highly negative sentiment detected ► Auto Escalate
     │
     │──► DRAFT Escalation Emails:
     │         ├── For Manager Review (human approves before sending to vendor)
     │         └── Auto-notify internally (no vendor-facing auto-send without approval)
     │
  📌 WHY IMPORTANT:
     Proactive escalation prevents breach and improves visibility.
     Vendor-facing escalations MUST require human approval.

     │         └── No response from vendor
     │                 └──► AUTO-CLOSE after policy-defined window
     │
     │──► AFTER CLOSURE - REOPEN DETECTION:
     │         │
     │         ├── Vendor replies after closure?
     │         │       │
     │         │       ├── YES ──► DETECT reply automatically
     │         │       │           ├── REOPEN existing ticket  OR
     │         │       │           └── LINK to prior ticket
     │         │       │               (avoid duplicate ticket creation)
     │         │       │
     │         │       └── NO  ──► Proceed to final audit step
     │         │
     │──► GENERATE Final Ticket Summary for:
     │         ├── Audit
     │         └── Reporting
     │
  📌 WHY IMPORTANT:
     Closure must be auditable and understandable for future reference.
     Reopen logic protects continuity and prevents fragmented conversations.
